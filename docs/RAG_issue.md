## 各轮迭代详情

### v1: 基线 (2026-05-22)

**方案**: sentence chunking (500字) + text-embedding-v3 + hybrid 检索

**为什么这样做**: 最简方案，验证端到端管线是否跑通。

**指标**: Recall@5=0.564, MRR=0.383, Faithfulness=1.000, Answer Relevance=0.527

**为什么不够**: 无法回答宏观问题（"乌合之众讲了什么"），chunk 太碎导致语义不完整。

---

### v2: 三层层级索引 (2026-05-23)

**方案**: L1 全书摘要 → L2 章节摘要 → L3 段落，宏观/微观自动路由

**为什么这样改**: 基线暴露了 chunk 级检索无法获得文档全局视角的根本局限。需要为宏观问题提供结构化摘要。

**指标变化**: vector MRR 0.330→0.494 (+50%), Answer Relevance 0.527→0.573 (+9%)

**为什么不够**: top-K=8 候选太少，混合检索的优势没发挥出来。

**相关文档**: [Issue #1](#issue-1-无法进行全局性文档检索与总结)

---

### v3: 扩大检索网 (2026-05-25)

**方案**: top_k 从 8 提升到 30

**为什么这样改**: 层级索引增加了摘要候选，但 top-K 太小导致好的候选被截断。增加候选池让 reranker 有更多选择。

**指标变化**: hybrid MRR 0.436→0.483 (+11%), Hit@8=0.800

**为什么不够**: 查询本身的语义匹配不够精准，向量检索命中率有天花板。

---

### v4: HyDE 假设文档检索 (2026-05-27)

**方案**: LLM 先生成假设答案，用假设答案替代原始查询做检索

**为什么这样改**: 用户查询往往简短且模糊，而假设答案与目标段落在语义空间中更接近 (Gao et al., 2022)。

**指标变化**: MRR 0.483→0.612 (+27%), Answer Relevance 0.580→0.607

**副作用**: HyDE 生成的假设文档会"稀释"原始查询的精确性，导致 Recall 略降。

**为什么不够**: sentence chunk 语义不完整的根本问题没解决，每个 chunk 只有 ~500 字，概念被切碎。

---

### v5: Sentence Window 实验 (2026-05-27) — 失败

**方案**: 单句嵌入 + Small-to-Big 检索 (嵌入用单句，返回用前后句子扩展)

**为什么尝试**: 学术论文中 Sentence Window 在某些数据集上表现优秀。

**结果**: 全面退化，Recall@5=0.361。

**失败原因**: 心理学文本的语义需要完整段落才能表达，单句嵌入丢失太多上下文。这个方案适用于事实密集型文本（新闻、法规），不适合论述性文本。

---

### v6: 段落级切分 + v4 Embedding (2026-05-28) — 当前方案

**方案**:
- `paragraph_chunker.py`: OCR 文本重建段落 (~1200字) + 链式 metadata
- text-embedding-v4 (DashScope 原生 SDK): text_type + instruct 参数
- 生成 prompt 严格区分核心/补充文档

**为什么这样改**:
1. **段落切分**: 之前所有方案都基于 sentence chunk，语义被切碎是根本瓶颈。v5 的失败验证了"需要更大粒度"的方向。
2. **v4 + text_type**: v3 的 OpenAI 兼容接口不支持 text_type/instruct，查询和文档在不同语义空间。v4 原生 SDK 解决了这个问题。
3. **严格 prompt**: 之前 Faithfulness=1.0 是因为 LLM 太保守（什么都不敢说），改 prompt 后 LLM 敢用检索到的内容了。

**指标变化**:
| 指标 | 旧基线 (chunk1024) | 新 | 变化 |
|------|-------------------|---|------|
| Recall@5 | 0.476 | 0.669 | +40.5% |
| Answer Relevance | 0.620 | 0.933 | +50.5% |
| Faithfulness | 0.933 | 0.920 | -1.4% |

---

### v6.1: Reranker 评估修复 (2026-05-28)

**问题**: `_get_retrieval_docs()` 将 `top_k=20` 传给 reranker，而候选也只有 20 条。reranker 判断 `len <= top_k` 后直接返回原序列。

**修复**: 改用 `pipeline.config.rerank_top_k=8` 做精选，20→8，reranker 真正生效。

**指标变化**:
| 指标 | hybrid (raw) | hybrid_reranked | 提升 |
|------|-------------|-----------------|------|
| MRR | 0.547 | **0.733** | +34% |
| Hit@1 | 0.467 | **0.733** | +57% |
| Recall@5 | 0.613 | **0.709** | +15.5% |

**当前最优指标** (hybrid_reranked): Recall@5=0.709, MRR=0.733, Hit@1=0.733, Faithfulness=0.873, Answer Relevance=0.953

## 迭代思路总结

整个迭代围绕一个核心矛盾展开：**语义完整性 vs 检索精度**。

```
chunk 越小 → 检索精度越高 (命中更精准)
chunk 越小 → 语义越不完整 (概念被切碎)
chunk 越大 → 语义越完整 (完整论述)
chunk 越大 → 检索精度越低 (噪声稀释相关性)
```

迭代过程就是在这个光谱上寻找最优点：

1. **v1** (500字sentence) → 太碎，宏观问题不行
2. **v2** (加层级索引) → 宏观问题解决了，但微观精度受限于 chunk 粒度
3. **v3-v4** (top_k, HyDE) → 在检索策略层面优化，但 chunk 粒度瓶颈没突破
4. **v5** (更小的chunk) → 走反了方向，全面退化
5. **v6** (1200字paragraph) → 找到平衡点：足够大保持语义完整，配合 reranker 过滤噪声

## 当前瓶颈：RAG 索引侧已到天花板

索引侧（切分、Embedding、检索、Rerank）的优化空间已有限。剩余问题转移到 **Agent 层**:

| 问题 | 根因 | 层级 |
|------|------|------|
| Faithfulness 87% (目标 95%+) | LLM 在信息不足时用自身知识补全 | Agent/生成 |
| 对比类问题回答不了 | 单次检索无法覆盖多角度 | Agent/查询 |
| 多跳推理准确率低 | 需要 Agent 多步推理串联信息 | Agent/推理 |

**下一阶段方向**: 从单趟 RAG pipeline 演进为多步 Agent 架构。详见下方。

## Issue #2: 上下文扩展引入噪声

**日期**: 2026-05-28
**严重程度**: 中
**状态**: 已缓解 (默认关闭，标记为补充文档)

### 问题描述

`expand_context()` 通过链式 metadata 拉取前后段落。测试发现扩展后的段落噪声比例高，导致 Faithfulness 从 0.933 跌到 0.613。

### 解决方案

1. 扩展段落标记 `is_expanded=True`
2. `format_context()` 分离核心/补充文档
3. 生成 prompt 指示 LLM 以核心文档为主
4. 默认关闭 (`context_window=0`)

---

## Issue #3: LLM Reranker 在评估中未生效

**日期**: 2026-05-28
**严重程度**: 高 (评估结果不准确)
**状态**: 已修复

### 问题描述

`hybrid_reranked` 与 `hybrid` 指标完全一致。根因: `_get_retrieval_docs()` 传 `top_k=20` 给 reranker，候选也是 20 条，reranker 直接返回原序列。

### 解决方案

改用 `pipeline.config.rerank_top_k=8` 做精选。修复后 MRR 从 0.547 提升到 0.733 (+34%)。
