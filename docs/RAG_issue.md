# RAG 系统问题记录

## Issue #1: 无法进行全局性文档检索与总结

**日期**: 2026-05-22  
**严重程度**: 中 (架构层面限制)  
**状态**: ✅ 已解决 (2026-05-23 层级索引方案)

### 问题描述

当前的 RAG 系统基于 chunk 级别检索：将文档切分为固定大小的 chunks（如 500 字符），用户的查询通过向量相似度 / BM25 匹配到最相关的 N 个 chunks，再由 LLM 基于这些 chunks 生成回答。

这种设计在处理**跨段落、全局性**问题时存在根本性局限——chunks 之间相互独立，无法获得文档的全貌。

### 复现用例

```bash
python main.py query "解读一下乌合之众这本书在讲什么" -v
```

### 实际输出

```
根据提供的上下文，文档中并未系统性地解读《乌合之众》全书的具体内容，但间接提到了该书的核心关注点。例如，文档2指出勒庞"有效地阐明了'个人在群体影响下，思想和感觉中道德约束与文明方式突然消失，原始冲动、幼稚行为和犯罪倾向的突然爆发'的实相"。这反映了该书对群体心理中个人行为变化的讨论。此外，文档1和4强调了该书对集体行为和社会心理学思考的巨大影响，以及它"从头到尾表现出一种对重要问题的敏感性"。然而，要全面解读该书内容，需要超出文档范围。因此，文档中未找到完整的解读信息。
```

### 问题分析

1. **Chunk 检索的固有局限**: 检索阶段（retrieval）从 1208 个 chunks 中只召回 top-K（默认 8）个最相关的片段，系统无法感知文档的整体结构和全部内容。

2. **上下文窗口浪费**: 召回的 8 个 chunks 总长度约 2276 字符，但其中包含大量相邻页面内容（如 `第 25 页提要...`、`第 26 页...`），缺少对全书核心观点的概括性描述。

3. **没有查询改写或分解**: 对于"解读一本书"这类高层级问题，系统没有将其分解为"该书的核心论点是什么"、"有哪些主要章节"、"结论是什么"等子问题。

4. **缺少文档级索引**: 当前只有 chunk 级的向量/BM25 索引，没有文档级的摘要索引来支持"这本书讲了什么"这类宏观问题。

### 可能的改进方向

| 方案 | 说明 | 复杂度 |
|------|------|--------|
| **文档摘要索引** | 对每个 PDF 先用 LLM 生成摘要，建立"文档 → 摘要"映射。全局性问题先查摘要，再按需深入 chunks | 中 |
| **层级索引 (Hierarchical Indexing)** | 建立 文档级 → 章节级 → 段落级 的多层索引，逐层缩小检索范围 | 高 |
| **查询分解 (Query Decomposition)** | 将复杂问题拆成子问题分别检索，汇总后回答 | 中 |
| **图谱索引 (Graph RAG)** | 将文档内容构建为知识图谱，支持跨 chunk 的关联推理 | 高 |
| **增大 top-K + 摘要 LLM** | 召回更多 chunks（如 top-50），用 LLM 做全局摘要后再回答问题 | 低 |
| **Map-Reduce 策略** | 对每个 chunk 分别做小回答，再汇总生成最终答案 | 中 |

### 相关上下文

- 检索模式: `hybrid` (向量 + BM25)
- Chunk 参数: size=500, overlap=80
- 检索 top-K: 8
- 精排后保留: 4
- 查询改写: 启用
- 嵌入维度: 1024 (DashScope text-embedding-v3)
- LLM: DeepSeek-V4-Flash

## Issue #1 解决方案：三层层级索引

**实施日期**: 2026-05-23

### 方案选型

采用**自建三层索引 + 摘要向量化**方案（详见 `docs/层级索引方案.md`），而非引入 GraphRAG 等重型框架。

### 架构

```
L1 文档级 (3 个节点)    — 全书摘要（LLM 基于各章摘要汇总生成）
L2 章节级 (~50 个节点)  — 每章标题 + LLM 摘要 + jieba 关键词
L3 段落级 (1208 chunks) — 原始段落（不变）
```

### 检索路由

| 查询类型 | 触发条件 | 检索策略 |
|----------|----------|----------|
| 宏观问题 | 含"总结/概括/全书/核心观点/讲了什么/框架/解读..."等关键词 | L1+L2 摘要优先 (k=6) → L3 段落补充 (k=6) → 去重合并 |
| 微观问题 | 默认 | 现有 BM25+向量+hybrid+rerank+rewrite 流程 |

### 实现文件

- `rag_agent/hierarchy.py` — 章节检测 + LLM 摘要生成 + 层级索引序列化
- `rag_agent/pipeline.py` — `ingest()` 中构建层级索引；`query()` 中路由检索
- `data/chunks/hierarchy_{timestamp}.json` — 层级关系落盘（36KB），可人工审查

### 效果对比

| 维度 | 扁平索引（改进前） | 层级索引（改进后） |
|------|:-----------:|:-----------:|
| 上下文来源 | 4 个碎片段落（2276 字符） | 6 条摘要 + 4 条段落（3573 字符） |
| 覆盖范围 | 仅 4 个 chunk 的局部片段 | L1 全书摘要 + 5 个 L2 章节摘要 |
| 回答质量 | ❌ "文档中未找到完整的解读信息" | ✅ 系统归纳核心论点 + 领袖 + 教育 + 分类 |
| 来源引用 | 模糊（"文档2"） | 具体（文档1、5 / 文档3、4、6） |
| 可回答性 | 0/10 | 7/10 |

## Issue #2: 层级索引的已知局限

**日期**: 2026-05-23  
**严重程度**: 低-中  
**状态**: 待优化

### 2.1 L3 段落偏边缘

宏观检索将 L3 段落（k=6）与 L1/L2 摘要（k=6）等权合并。但 L3 召回的常是附录/书评/后记（如 189-242 页的第三方评论），而非正文核心内容。

**改进方向**: 宏观问题降低 L3 权重（仅 k=2），或仅在摘要覆盖不足时补充。

### 2.2 宏观判断靠关键词

`is_macro_query()` 用 16 个关键词做字符串匹配，漏判率高。如 "这本书想表达什么" 不会被识别为宏观问题。

**改进方向**: 用 LLM 做一次轻量二分类路由（"这个问题需要全局理解吗？是/否"），成本 ~10 tokens。

### 2.3 人人都该懂的心理学 章节检测偏少

该书只有数字编号（`1　XXX`），无 "第X章" 标记，正则未覆盖，只检测到 7 节。

**改进方向**: 增加 `r"^\d+[\s　]+"` 模式识别阿拉伯数字章节。但当前 7 节已覆盖全书主要内容，优先级可降低。

### 2.4 来源显示不友好

来源列表中显示 `hierarchy` 而非 `乌合之众：大众心理研究（全书摘要）`。

**改进方向**: 在 `query()` 返回前做来源名称映射。

### 2.5 摘要不带 rerank

L1/L2 摘要检索直接用 `similarity_search`，未经过 BM25 融合和 LLM 精排。宏观问题对排序精度要求不如微观问题高，当前影响不大。

### 索引规模

- L3 段落: 1208 chunks
- L1/L2 摘要: 36 条
- 向量库总量: 1244 条
- Chunk 参数: size=500, overlap=80
- Embedding: DashScope text-embedding-v3, dim=1024
- LLM: DeepSeek-V4-Flash
