# Agent 层优化路线图

**日期**: 2026-05-29
**状态**: P0 + P1 + P2 已完成，评估中

已从单趟 RAG Pipeline 演进为 LangGraph 多步 Agent 架构。当前指标 (Faithfulness=0.903, Answer Relevance=0.913) 中，Faithfulness 较基线提升 5.2%，但距商用 95% 门槛仍有差距。

---

## 当前问题诊断

### Faithfulness 87% 的根因

| 问题类型 | Faithfulness | 原因 |
|---------|-------------|------|
| 事实/定义/总结 | 0.95+ | 信息充分，LLM 直接引用 |
| 多跳推理 | 0.50 | 上下文有部分信息，LLM 用自身知识补全空缺 |
| 对比分析 | N/A (拒绝回答) | 检索只命中单方面信息，无法对比 |

具体表现：
- 编造不存在的数据 (如"85%的疗效")
- 引用不存在的文档编号
- 超出上下文范围的细节"脑补"
- 文档中实际有相关信息但检索未命中

### 核心矛盾

当前是 **单趟 pipeline**: 检索一次 → 生成一次 → 结束。

这导致：
1. **检索遗漏无法补救**: 如果第一次检索没找到关键信息，LLM 只能用自身知识补全
2. **复杂查询无法拆解**: "对比焦虑和恐惧的区别"需要分别检索两个概念再做对比，单次检索做不到
3. **没有自我校验**: 生成的回答是否真的有据可依，没有检查环节

---

## 优化方案

### 方案 1: 两阶段生成 (Claim Verification)

**目标**: Faithfulness 87% → 95%+

**原理**: 把生成拆成两步：先提取事实，再基于事实组织回答。中间加一层校验。

```
当前:  Context → LLM → Answer (一步到位，幻觉混在其中)

改进:  Context → LLM提取事实 → 校验(事实是否在Context中) → LLM组织回答
                        ↑                          ↓
                      过滤掉无据可依的事实      只使用验证通过的事实
```

**实现要点**:
- 第一步: LLM 从 context 提取与问题相关的声明 (claims)，每个 claim 标注来源
- 校验: 检查每个 claim 是否能在 context 中找到对应文本
- 第二步: 只基于验证通过的 claims 组织最终回答

**预期收益**: Faithfulness +8-10%，因为未验证的信息在中间层被过滤掉了

---

### 方案 2: 查询分解 (Query Decomposition)

**目标**: 对比/多跳类问题的 Answer Relevance 提升

**原理**: 复杂问题拆成子问题，每个子问题独立检索，结果合并后生成。

```
当前:  "CBT核心技术有哪些？如何应用于焦虑症？"
       → 一次检索 → 可能只命中"技术列表"或"应用案例"，不全

改进:  拆解为 ["CBT核心技术有哪些", "CBT如何治疗焦虑症", "暴露疗法的应用"]
       → 每个子问题独立检索 → 结果合并 → 完整回答
```

**实现要点**:
- LLM 判断问题是否需要分解（简单事实查询不需要）
- 分解后的子问题独立走检索管线
- 合并检索结果时去重、排序
- 生成时综合所有子问题的上下文

**预期收益**: 对比/多跳类 Answer Relevance 显著提升，Recall 也会改善

---

### 方案 3: 置信度感知 (Confidence-Aware Response)

**目标**: 降低误导用户的风险

**原理**: 评估检索质量和回答置信度，低置信度时主动降低肯定程度。

```
当前:  所有回答语气一致，用户无法区分高/低置信度

改进:  高置信度 → "根据文档，灾难化思维是指..."
       低置信度 → "根据现有文档，可能有以下几种解释，但信息不够完整..."
       无信息   → "文档中未找到相关信息"
```

**实现要点**:
- 检索质量评估: top-K 结果的相似度分数分布、reranker 评分分布
- 回答置信度: 基于检索质量 + context 覆盖度打分
- 输出策略: 置信度低于阈值时附加免责声明

**预期收益**: 用户体验提升，错误回答的风险降低

## Agent 架构演进方向

当前架构:
```
Query → Retrieve → Rerank → Generate → Answer
```

目标架构:
```
Query → [Query Understanding]
            │
            ├→ 简单查询 → Retrieve → Rerank → Generate
            │
            ├→ 复杂查询 → Decompose → [子查询 × Retrieve × Rerank] → Merge → Generate
            │
            └→ 宏观查询 → Hierarchy Route → ...

Generate = Extract Claims → Verify → Compose Answer (置信度标注)
```

关键变化: 从**线性 pipeline** 变为**有条件分支的 Agent**，核心增加的能力:
1. **查询理解**: 判断问题类型和复杂度
2. **多步检索**: 按需拆解和补充检索
3. **自我校验**: 生成后验证回答质量
4. **置信度输出**: 让用户知道答案的可靠程度
